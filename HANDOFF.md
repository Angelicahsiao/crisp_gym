## 1. Non-negotiable conventions (breaking any of these silently corrupts data/training)

### 1.1 Pose layout (rot6d, UMI convention)
- Cartesian obs: `[x, y, z, r00, r01, r02, r10, r11, r12]` â€” 9D.
  The 6 rotation numbers are the **first two ROWS** of the 3x3 rotation matrix,
  flattened row-major (UMI / pytorch3d convention â€” NOT columns).
- Action: the same 9D pose + 1 gripper = 10D.
- Identity rotation in rot6d = `[1,0,0, 0,1,0]`. Identity pose = `[0,0,0, 1,0,0, 0,1,0]`.
- Decoding uses Gram-Schmidt (`rot6d_to_mat` in
  `crisp_gym/scripts/lerobot_relative_pose.py`) â€” network outputs are not
  perfectly orthonormal.
- Set via `orientation_representation: "rotation_6d"` (enum member `ROTATION_6D`
  in `crisp_py/utils/geometry.py`). Do not switch representations without
  re-recording or migrating existing datasets.


- Canonical in-package implementation: `crisp_gym/util/rot6d.py`. The two
  standalone scripts keep local copies by design (no crisp imports).
- DEPLOY INVARIANT: every action of a relative-action chunk is composed
  against the pose captured at the chunk's OBSERVATION time
  (`RemotePolicy._chunk_base`), never the live execution-time pose â€”
  per-step composition re-adds executed motion and compounds within the
  chunk. Regression-tested in tests/test_pose_math.py.

### 1.2 What is stored on disk vs what the model sees
- **On disk (Parquet, LeRobot format): ABSOLUTE poses.** Never store relative
  poses in the dataset. `observation.state.cartesian` = absolute TCP pose in
  the recording world frame; `action[t]` = absolute TCP pose at t+1
  (1-step lookahead, see `make_record_fn` in
  `crisp_gym/record/record_functions.py`).
- **At training time**: `crisp_gym/scripts/lerobot_relative_pose.py`
  (`RelativePoseDataset`) converts to relative inside `__getitem__`, AFTER
  LeRobot's `delta_timestamps` temporal stacking:
  `T_rel = T_current^-1 @ T_target` with the base = LAST frame of the obs
  window (the current frame). This mirrors UMI `umi_dataset.py`
  (`base_pose_mat = pose_mat[-1]`, `pose_rep='relative'`).
- The wrapper converts `observation.state.cartesian`, `action`, AND the
  CONCATENATED `observation.state` (pose dims only; gripper/extras pass
  through), then APPENDS the wrt-start rot6d to `observation.state` â€” final
  model input `[rel_pose9, gripper1(, extras...), rot_wrt_start6]` = UMI's
  16-D low-dim state (position-wrt-start deliberately excluded, as in UMI).
  `observation.state` is what lerobot-0.4.4 policies actually consume
  (OBS_STATE queue); the wrapper also widens its `meta.features` entry so the
  policy input layer is sized correctly. THREE checkpoint generations exist:
    1. absolute (pre-conversion wrapper; no pose_repr.json) â€” ABSOLUTE 10-D
       state input (relative actions were always correct);
    2. relative 10-D (state converted, no wrt-start appended);
    3. relative_wrt_start 16-D (current â€” `state_includes_wrt_start: true`
       in pose_repr.json).
  Deployment (`relative_lerobot_policy`, and later the remote server) keys
  its obs conversion off the stamp (`state_input: auto`) â€” never deploy a
  checkpoint with another generation's obs semantics. Deploy-side wrt-start:
  computed from ABSOLUTE poses FIRST (episode start = first obs after reset,
  NOISE OFF), then the relative conversion rewrites dims 0-8 only.
- Consequences the next developer must preserve:
  - Current obs frame becomes identity (pos zeros + rot6d [1,0,0,0,1,0]) â€” this
    is CORRECT, not a bug. Information lives in the t-1 frame and images.
  - Relative conversion CANNOT be pre-baked into Parquet per-row: obs[t-1] must
    be relative to obs[t], which depends on which frames the sampler pairs.
  - Normalization stats MUST be recomputed on relative values
    (`recompute_relative_stats`) â€” LeRobot's stats are computed on the absolute
    disk values and are wrong for the converted tensors.

### 1.3 wrt_start observation
- `observation.state.rot_wrt_start` (rot6d of each obs frame relative to the
  episode-start pose) is generated BY THE WRAPPER at load time, rotation-only,
  with additive noise scale 0.05 on the start pose (UMI's HACK in
  umi_dataset.py:314). It is NOT in `dataset.meta.features`; policy configs
  must declare it explicitly as an input feature.
- UMI records the full start pose but feeds only rotation to the policy
  (position-wrt-start is deliberately commented out in UMI â€” do not "fix" this
  by adding position; it would break spatial generalization).
- Since the 16-D state change (Â§1.2), the wrt-start rot6d ALSO gets appended
  into `observation.state` â€” that is how it actually reaches a lerobot-0.4.4
  policy (the standalone `rot_wrt_start` key is not consumable there). The
  standalone key is still emitted for stats/debugging.

### 1.4 Frames and calibration
- `T_tcp = tx_world_correction @ T_optitrack_body @ tx_body_tcp`
  (left = world-side correction, right = tool-side offset; see
  `UmiHandheldEnv._pose_msg_to_array`). Both currently identity.
- Because training uses relative pose, the WORLD frame cancels â€” no
  OptiTrack-to-robot-base calibration is needed, and this was verified
  numerically (world-frame invariance test).
- What does NOT cancel: `tx_body_tcp` (an axis/origin error here conjugates
  every relative motion: T_rel = E^-1 T^-1 T' E). The handheld TCP frame must
  match the robot URDF TCP convention (for Franka `fr3_hand_tcp`: z = approach
  through fingertips, y = jaw axis; origin between fingertips).
- Deployment composition: `T_cmd = T_current_robot_tcp @ T_rel`, using the TCP
  pose captured AT OBSERVATION TIME (not at action-receive time).

### 1.5 Gripper
- Stored normalized to [0,1] via `max_gripper_width` (0.09 m default) â€” same
  scaling must be used for handheld recording AND robot deployment.
- Gripper dim passes through all pose conversions untouched.

### 1.6 torchcodec + forked DataLoader workers â€” TRAINING GOTCHA (SOLVED in-script)
- Symptom: training via `lerobot_relative_pose.py` with the default
  `torchcodec` backend dies at step 0 in a DataLoader **worker** with
  `RuntimeError: Could not push packet to decoder: Invalid data found`.
- ROOT CAUSE (verified): NOT the videos and NOT a seek bug. The videos decode
  fine â€” plain `lerobot-train` works, and `--num_workers=0` works. The wrapper's
  `recompute_relative_stats` ran in the MAIN process and opened torchcodec
  decoders; the DataLoader then FORKS workers, which inherit that torchcodec
  state broken and crash. torchcodec decoders are not fork-safe.
- FIX (in `lerobot_relative_pose.py`, `recompute_relative_stats`): the stats
  keys are all low-dim poses, so the pass temporarily no-ops the dataset's
  `_query_videos` â€” no decoder is opened in the main process, workers make
  fresh ones, torchcodec runs at full `--num_workers`. DO NOT reintroduce video
  decoding into that pass.
- Fallbacks if a future change breaks this: `--dataset.video_backend=pyav`
  (correct, slower) or `--num_workers=0` (no fork, slow).

## 3. Verification: how to know a pose-math change is correct

Re-run the wrapper test (recreate if missing â€” historical copy described here):
synthetic SE(3) trajectory -> `RelativePoseDataset.convert_item` -> assert:
1. rot6d<->matrix round-trip exact on random rotations.
2. Current obs frame -> identity pose.
3. Round-trip `T_base @ T_rel == T_abs` for every obs & action step (compare as
   MATRICES, never as raw 6D/9D vectors â€” representation wrap hides errors).
4. Gripper dim bit-exact through conversion.
5. World-frame invariance: left-multiply ALL absolute poses by one random
   rigid transform -> relative outputs must be unchanged (this is the property
   that makes handheld->robot transfer work; if this test fails, deployment
   breaks even if everything else passes).
6. wrt_start: shape (n_obs,6), rotation-only, start-pose noise perturbs only
   this key.

A quick recorded-data sanity check: action rows `a[3:6]`,`a[6:9]` must have
norm ~1.0 and dot ~0.0 (they are rows of a real rotation matrix).

## 4. Roadmap / pending work (agreed with the owner; do not re-litigate)

1. **WebSocket inference client** (`crisp_gym/policy/remote_policy.py`,
   `@register_policy("remote")`): msgpack-numpy over websocket (openpi-style).
   NO torch/lerobot/rclpy imports in the client. Client sends ABSOLUTE poses +
   images (uint8 HWC) + task string; SERVER does the relative conversion
   (must reuse/verbatim-match `RelativePoseDataset.convert_item`) and
   normalization. Client composes returned relative actions with the TCP pose
   captured at obs time. Chunked: server returns `chunk_size` steps, client
   executes `n_action_steps` then re-queries. Handshake metadata schema was
   specified in session notes â€” key fields: obs keys/shapes/dtypes, n_obs_steps,
   pose_convention, action layout/pose_repr/chunking/dt, gripper semantics,
   normalization="server_side", reset message support.
   **The server itself is NOT this project's job** â€” the owner builds it; we
   provide the client + the metadata contract.
2. **(IMPLEMENTED) `pose_repr.json` stamping**: `lerobot_relative_pose.py`
   writes it into the training output dir (pose layout, relative convention,
   rot6d convention, fps, window sizes, and per-key `observation.state`
   semantics â€” the field deployment keys its obs conversion off, Â§1.2).

## 5. Process rules for AI sessions on these repos

- Follow `CLAUDE.md`: for the first request of a session, present a plan,
  change nothing beyond what is explicitly approved, and ask before proceeding.
- Branch: develop on the designated `claude/...` branch; never push elsewhere
  without explicit permission. The owner merges PRs via the GitHub website.
- Before assessing "what exists", check ALL remote branches
  (`git ls-remote --heads origin`) in BOTH repos â€” parallel work streams
  (UR7e tuning, lerobot pinning, DG3F gripper) have appeared on side branches
  before and conflicted semantically (e.g. docker-compose DDS settings).
- docker-compose DDS settings are machine-specific and fought over once
  already: keep `ROS_DOMAIN_ID=1`, `NETWORK_INTERFACE=enp0s31f6`, generated
  CycloneDDS config, and the container startup chain
  `gen_cyclone_config.sh -> setup_lerobot.sh -> pixi install`.
- When editing `lerobot_features.py`, preserve BOTH: rotation-rep-aware
  dims/names AND the lerobot v0.4/v0.5 import/version/fps compatibility.
- When touching pose math anywhere, re-run the verification suite in section 3
  before pushing. Compare rotations as matrices.
- The user typically records on a ROS2 Humble/Python 3.11 machine and trains on
  a separate GPU PC with lerobot 0.4.4 â€” keep the training/inference files free
  of crisp/ROS imports.
